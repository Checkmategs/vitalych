# Design: MVP шаблонов техдокументации (ГОСТ)

**Date:** 2026-07-23  
**Status:** Approved (brainstorming)  
**Scope:** Этап A (шаблоны) → этап B (CLI). Веб-UI вне scope.

## Goal

Дать единый набор переменных проекта и два самодостаточных Jinja2-шаблона Markdown:

1. **Техническое задание (ТЗ на АС)** — по ГОСТ 34.602-2020  
2. **Пояснительная записка (ПЗ)** к эскизному/техническому проекту — по ГОСТ Р 59795—2021 п. 5.2  

Затем CLI: YAML → отрендеренные `.md` (DOCX — позже).

## Non-goals (MVP)

- Веб-редактор / multi-user / версии  
- PDF, рамка ГОСТ, ИИ  
- `{% include %}` и библиотека фрагментов  
- Полная валидация «все пункты ГОСТ заполнены»

## Architecture

```text
Vitalych/
  templates/
    tz.md.j2
    pz.md.j2
  data/
    project.example.yaml   # единый набор переменных
  src/                     # этап B
    render.py
  out/
  docs/superpowers/specs/
```

- Один `project.yaml` питает оба шаблона.  
- Нет дублей ключей вида `tz.*` / `pz.*` для одних и тех же сущностей; блок `pz` — только поля, нужные ПЗ и не покрытые общими разделами.  
- Шаблоны самодостаточны (без include).

## Shared data model (`project.yaml`)

| Блок | Назначение | Потребители |
|------|------------|-------------|
| `system` | имя АС, обозначение, шифр | ТЗ §1, ПЗ §1 |
| `parties` | заказчик, разработчик, участники | оба |
| `basis` | документы-основания | оба |
| `schedule` | сроки, стадии | оба |
| `funding` | финансирование | ТЗ §1 |
| `goals` | цели, назначение, критерии | ТЗ §2, ПЗ |
| `object` | объект автоматизации, среда | ТЗ §3, ПЗ процессы |
| `requirements` | структура, функции, обеспечения, общие ТТ | ТЗ §4; отсылки в ПЗ «решения» |
| `works` | этапы и содержание работ | ТЗ §5 |
| `development` | порядок разработки | ТЗ §6 |
| `acceptance` | контроль и приёмка | ТЗ §7 |
| `prep` | подготовка объекта к вводу | ТЗ §8, ПЗ §4 |
| `docs` | требования к документированию | ТЗ §9 |
| `sources` | источники разработки (`entries[]`) | ТЗ §10 |
| `pz` | ТБ, НТД, НИР, очереди создания и т.п. | ПЗ |
| `meta.absent_sections` | коды разделов ТЗ без требований | ТЗ |

### Правило пустых разделов ТЗ

Если код раздела (например `requirements`, `funding`) есть в `meta.absent_sections`, шаблон **сохраняет заголовок** и выводит: «Требования отсутствуют.»

Коды: `general` не используют (обязательный каркас всегда есть); типичные: `requirements`, `works`, `development`, `acceptance`, `prep`, `docs`, `sources`, `funding`.

## Template: ТЗ (`tz.md.j2`)

Обязательные разделы (порядок ГОСТ 34.602 п. 4.1 / 5.1):

1. Общие сведения  
2. Цели и назначение создания автоматизированной системы  
3. Характеристика объектов автоматизации  
4. Требования к автоматизированной системе  
5. Состав и содержание работ по созданию автоматизированной системы  
6. Порядок разработки автоматизированной системы  
7. Порядок контроля и приёмки автоматизированной системы  
8. Требования к составу и содержанию работ по подготовке объекта автоматизации к вводу автоматизированной системы в действие  
9. Требования к документированию  
10. Источники разработки  

Подразделы §2 и §4 — по стандарту (цели/назначение; структура/функции/обеспечения/общие ТТ).  
Недостающие данные: italic-подсказка `_заполнить: …_`.

## Template: ПЗ (`pz.md.j2`)

Разделы (ГОСТ Р 59795 п. 5.2.1):

1. Общие положения  
2. Описание процессов деятельности объекта автоматизации  
3. Основные технические решения  
4. Мероприятия по подготовке объекта автоматизации к вводу АС в действие  

Содержательные пункты — чеклисты из п. 5.2.2–5.2.5 с подстановкой из YAML.

## Error handling (B)

| Случай | Поведение |
|--------|-----------|
| Нет data/шаблона | exit ≠ 0, сообщение в stderr |
| Неизвестный ключ в шаблоне | Jinja StrictUndefined → fail |
| Пустой `absent_sections` | все разделы рендерятся из данных |
